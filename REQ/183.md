# R-#183: Replace Rails Profile Backend with Next.js API

## Objective
Replace the Rails-dependent profile CRUD with portable Next.js API routes that read/write directly to PostgreSQL, eliminating the Rails dependency for user profile management.

## Background
The profile page currently depends on 4 Rails endpoints:
- `NEXT_PUBLIC_API_USERS` — GET users (filtered by walletAddress)
- `NEXT_PUBLIC_API_UPDATE_USER` — PATCH user profile
- `NEXT_PUBLIC_API_COUNTRIES` — GET countries list
- `NEXT_PUBLIC_API_SHOW_USER` — GET single user

New columns from R-#152 (`whatsapp`, `telegram`, `church_relationship`, `church_id`, `place_of_worship`, etc.) are not persisted because the Rails controller does not whitelist them, and Rails is being phased out.

---

## Option 1: Add fields to Rails backend

**Pros:**
- Minimal code change (~5 lines in Rails controller)
- Existing auth already works
- Zero risk of breaking existing flows

**Cons:**
- Rails is in transition — adds maintenance burden to a dying component
- Not portable — tied to Rails MSIP framework
- Every new column requires Rails changes
- Delays the inevitable migration

**Effort:** 1h

---

## Option 2: Next.js API for profile (recommended)

Replace all 4 Rails endpoints with Next.js routes that read/write `usuario` directly via Kysely.

### MVP Scope

| Rails endpoint | Next.js replacement | Notes |
|---------------|-------------------|-------|
| `GET /usuarios.json?filtro[walletAddress]=X` | `GET /api/profile` | Single endpoint, auth via token |
| `POST /usuarios/actualiza_mi_usuario` | `PATCH /api/profile` | Update authenticated user's fields |
| `GET /admin/paises.json` | `GET /api/countries` | Already have `/api/religions` — same pattern |
| `GET /usuarios/:id.json` | `GET /api/user/[id]` | Already exists |

### Profile API design

```
GET  /api/profile?walletAddress=X&token=Y
  → Returns usuario record with all columns

PATCH /api/profile?walletAddress=X&token=Y
  → Updates allowed fields: nombre, nusuario, email, religion_id, pais_id,
    church_relationship, whatsapp, telegram, place_of_worship, church_id,
    department_id, municipality_id, city_id
  → Returns updated record
```

### Auth
Uses existing `authenticateUser(walletAddress, token)` — same pattern as scholarship, church, cluster APIs.

### What this enables immediately
- Profile save persists ALL fields (whatsapp, telegram, church_relationship, etc.)
- Profile fetch returns ALL fields
- No Rails dependency for profile

### Phase 2: Extract to @pasosdejesus/m

Once stable, extract the profile CRUD + auth layer into a portable module named `usuario` (matching the existing table name `usuario` and the MSIP Spanish naming convention):

```
@pasosdejesus/m/usuario
  ├── api/
  │   ├── route.ts          # GET + PATCH /api/usuario
  │   └── [id]/route.ts     # GET /api/usuario/[id]
  ├── types.ts              # Usuario interface
  └── utils.ts              # Field whitelist, validation
```

This follows the same pattern as MSIP's engine modules (WBA, credentials) — portable, testable, framework-agnostic.

### Rails Context

The current Rails `usuarios_controller#actualiza_mi_usuario` (line 106-132) only updates:
```ruby
u.update({
  nusuario: params["nusuario"],
  nombre: params["nombre"],
  email: params["email"],
  religion_id: params["religion_id"],
  pais_id: params["pais_id"],
})
```

New columns (`church_relationship`, `whatsapp`, `telegram`, `place_of_worship`, `church_id`, 
`department_id`, `municipality_id`, `city_id`, `id_photo_front`, `id_photo_back`) 
are NOT in the whitelist and will never be persisted through Rails.

The Rails index endpoint (`UsuariosController#index_otros_formatos`) renders JSON with the 
columns defined in `atributos_index` (line 10-22), which also doesn't include the new columns.

The `usuario.rb` model uses `cor1440_gen` concerns and `Msip::ModelosController` as base — 
tightly coupled to the MSIP framework.

### Future: MSIP user layer
Eventually, all user management (registration, profile, verification, scoring) could be a standalone engine in `@pasosdejesus/m/usuario`, similar to how `cor1440_gen` handles courses. This would make `learn.tg` a thin Next.js shell over portable engines.

### Comparison

| Aspect | Option 1 (Rails) | Option 2 (Next.js) |
|--------|-----------------|-------------------|
| Effort | 1h | 3-4h |
| Risk | Low | Medium (new code) |
| Portability | None | High |
| Future-proof | No | Yes |
| Deletes Rails dependency | No | Partial |
| New fields auto-supported | No | Yes (read/write all columns) |

### Recommendation
**Option 2 MVP** — replace profile CRUD with Next.js routes. Low risk because the pattern already exists (`/api/religions`, `/api/user/[id]`, church/cluster APIs). This is the last step before the profile page is fully functional with R-#152 fields.
