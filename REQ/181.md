# R-#181: Delete Verified Data Button in User Profile

## Objective

Add a **"Delete Verified Data"** button to the user profile page (`app/[lang]/profile/page.tsx`) that removes all verified fields from the user's record and resets their profile score, as described in the Privacy Policy (R-#180).

## Motivation

- **Privacy Policy (R-#180)** now includes a clause stating users can delete verified data immediately via a button on their profile page.
- **R-#152** introduced verified fields (`verified_whatsapp`, `verified_telegram`, `verified_department_id`, `verified_municipality_id`, `verified_city_id`, `verified_place_of_worship`) that increase profile score.
- Users should have self-service control over their verified data without needing to contact pdJ staff.

## Dependencies

- R-#152 (Church/Cluster Profiles) — verified fields and profile score logic
- R-#180 (Privacy Policy) — the policy text that references this button

## Design

### Button Placement

The button should be placed in the profile page, in a dedicated section for verified data management:

- **Location:** Below the profile score display, in a "Verified Data" section
- **Label:** "Delete Verified Data" / "Eliminar Información Verificada"
- **Style:** Destructive variant (red) with a warning icon
- **Confirmation:** A modal dialog asking the user to confirm the action

### Confirmation Dialog

### Backend API

**Endpoint:** `DELETE /api/user/verified-data`

**Authentication:** JWT (wallet + token)

**Process:**

1. Verify user is authenticated (JWT from `billetera_usuario.token`).
2. Delete all `verified_*` fields from `usuario` table:
   - `verified_whatsapp` → `NULL`
   - `verified_telegram` → `NULL`
   - `verified_department_id` → `NULL`
   - `verified_municipality_id` → `NULL`
   - `verified_city_id` → `NULL`
   - `verified_place_of_worship` → `NULL`
   - `id_photo_verified` → `FALSE`
3. Recalculate profile score (`updateProfileScore` from `lib/gd-utils.ts`).
4. Log event: `verified_data_deleted` in `userevent`.
5. Return success with new profile score.

### Database

No schema changes needed — the `verified_*` fields already exist from R-#152.

### User Event

```sql
-- Record deletion for audit
INSERT INTO userevent (usuario_id, event_type, event_data, created_at)
VALUES (?, 'verified_data_deleted', jsonb_build_object('timestamp', NOW()), NOW());
```

## API Endpoint

### `DELETE /api/user/verified-data`

**Request Headers:**

```
Authorization: Bearer <jwt>
```

**Response (Success):**

```json
{
  "success": true,
  "profileScore": 0,
  "message": "Verified data deleted successfully"
}
```

**Response (Error):**

```json
{
  "success": false,
  "error": "Unauthorized"
}
```

## Frontend Changes

### 1. Confirmation Dialog Component

- Uses shadcn/ui `Dialog`, `Button`, `Input` components

- Calls `DELETE /api/user/verified-data` on confirmation
- Shows success toast after deletion
