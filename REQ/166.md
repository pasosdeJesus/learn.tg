# E-#166: Post-Quantum Key Management Module

Upgrade the encryption system (#153) to be quantum-resistant by replacing the ECDSA-derived DEK with a SPHINCS+ keypair managed transparently within the learn.tg application.

## Dependencies
- R-#153 (Encrypted Profile Data — the system being hardened)
- Existing SIWE authentication (wallet remains for auth only)

## Motivation

The current system derives the user DEK from `personal_sign` (ECDSA over secp256k1). ECDSA is not post-quantum safe — Shor's algorithm can recover the private key and reconstruct the DEK. While AES-256-GCM protects the data itself, the DEK derivation path is the weak link.

Migrating to SPHINCS+ (NIST-standardized, 2024) fixes this at the encryption layer without waiting for Ethereum to migrate. Since the DEK is used off-chain, SPHINCS+'s large signature sizes (~8-49 KB) are acceptable.

## Design

### Key Generation (Transparent to Pastor)

```
Pastor signs up / first use of learn.tg
        │
        ▼
learn.tg app generates SPHINCS+ keypair (browser, WebAssembly)
        │
        ▼
Private key → encrypted with DEK (from personal_sign, current #153)
Public key  → stored in billetera_usuario.spx_public_key
        │
        ▼
Pastor never sees or manages PQ keys directly
```

### DEK Encryption (Replaces ECDSA step)

```
Server generates DEK (crypto.randomBytes 32)
        │
        ▼
DEK encrypted with pastor's SPHINCS+ public key  → encrypted_dek_user
DEK encrypted with pdJ's SPHINCS+ public key      → encrypted_dek_pdj
        │
        ▼
Both stored in usuario table
```

Pastor decrypts: learn.tg app uses SPHINCS+ private key (decrypted from DEK-protected local storage) to decrypt `encrypted_dek_user` → gets DEK → decrypts data.

### Recovery (12-word Phrase)

The pastor must be able to recover their PQ keys if they lose their device or browser data.

```
On key generation:
  1. Generate 12-word recovery phrase (BIP39 entropy → seed)
  2. Derive SPHINCS+ keypair from seed (deterministic)
  3. Show recovery phrase ONCE to pastor — "write this down"
  4. Pastor stores phrase offline (paper, same as wallet seed)

On recovery:
  1. Pastor opens learn.tg on new device
  2. Enters 12-word recovery phrase
  3. SPHINCS+ keypair re-derived from seed
  4. Access restored
```

**Security:** The recovery phrase never leaves the pastor's device. pdJ never sees it. It is NOT stored in any database. If the pastor loses it, recovery is impossible (same trust model as crypto wallets).

### pdJ Keys

pdJ also gets a SPHINCS+ keypair:
- Generated at deploy time from `PRIVATE_KEY` seed
- Stored in environment variables:
  - `LEARNTG_SPX_PRIVATE_KEY`
  - `NEXT_PUBLIC_LEARNTG_SPX_PUBLIC_KEY`
- Used server-side only (no browser wallet limitation for pdJ)

## Schema Changes

```sql
ALTER TABLE billetera_usuario ADD COLUMN spx_public_key TEXT;
ALTER TABLE usuario ADD COLUMN encrypted_dek_user_spx TEXT;
ALTER TABLE usuario ADD COLUMN encrypted_dek_pdj_spx TEXT;
ALTER TABLE usuario ADD COLUMN encryption_version INTEGER DEFAULT 2;
```

`encryption_version = 2` indicates SPHINCS+ encryption active. Version 1 = legacy ECDSA (for migration). If both are present, SPHINCS+ takes priority.

## Migration Path

```
Existing user (encryption_version = 1):
  → Uses ECDSA flow (current #153)

User generates SPHINCS+ keypair:
  → Data re-encrypted with SPHINCS+ DEK
  → encryption_version = 2
  → Both versions coexist during migration window

All users migrated:
  → ECDSA columns can be dropped
```

## Technical Notes

- **SPHINCS+ implementation:** `@openpgp/spx` or WASM-compiled `liboqs`
- **Signature size:** ~8 KB (SPHINCS+-128s) or ~49 KB (SPHINCS+-128f). Acceptable for off-chain DEK encryption (one-time per session).
- **Key generation:** ~50-200ms in browser (WebAssembly). Done once, not per session.
- **BIP39 compatibility:** Recovery phrase uses standard wordlist — pastors already understand this model from crypto wallets.

## Acceptance Criteria

- [ ] SPHINCS+ keypair generated transparently on first use
- [ ] Private key encrypted with DEK and stored locally only
- [ ] 12-word recovery phrase shown once with clear "write this down" instructions
- [ ] Recovery works: enter phrase → re-derive keys → decrypt data
- [ ] pdJ SPHINCS+ keypair generated at deploy time
- [ ] Migration path: legacy ECDSA and SPHINCS+ coexist
- [ ] `encryption_version = 2` for SPHINCS+-encrypted data
- [ ] No recovery phrase stored in database or logs
