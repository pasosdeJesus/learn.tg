# R-#173: "Partial login" after navigating away from landing page

## Symptom
After connecting OneKey wallet on `https://learn.tg/` and signing SIWE, clicking
English shows "Partial login. Please disconnect your wallet and connect and sign
again." on `/en`.

## Reproduction
1. Open new browser tab → `https://learn.tg`
2. Activate OneKey extension → reload page
3. Click Connect Wallet → confirm connection → sign SIWE
4. Click English → `/en` shows "Partial login"

E2E test with Puppeteer + simulated SIWE does NOT reproduce — only happens
with real OneKey wallet extension.

## Root Cause Analysis

### Causa raíz confirmada

**NextAuth bug conocido:** `useSession()` retorna `status: 'unauthenticated'` tras navegación cliente (`<Link>`, `router.push()`), incluso con cookie `next-auth.session-token` válida. Documentado en:

- [NextAuth discussion #5719](https://github.com/nextauthjs/next-auth/discussions/5719) — 37 participantes, 27 comentarios
- [NextAuth issue #9504](https://github.com/nextauthjs/next-auth/issues/9504)

**Mecanismo:** El `SessionProvider` de NextAuth no re-fetchea la sesión automáticamente en navegación cliente. Su polling interno (`refetchInterval`) solo dispara periódicamente, no en `router.push()`. Cuando el componente se monta en la nueva ruta, `useSession()` recibe el estado inicial del contexto (`null`) antes de que el poll se ejecute.

### Por qué el E2E no lo reproduce
El test de Puppeteer simula SIWE vía POST directo a `/api/auth/callback/credentials` y verifica la cookie. Pero no monta el `SessionProvider` de NextAuth ni ejecuta `useSession()` en el cliente React — solo verifica que la cookie existe. El bug está en la capa cliente de NextAuth, no en el servidor.

### Por qué con OneKey real sí ocurre
Con wallet real, la secuencia es:
1. Wagmi se reconecta en mount (por defecto) → `address` poblado
2. `RainbowKitSiweNextAuthProvider` detecta wallet conectada → dispara SIWE
3. SIWE completa → cookie de sesión seteada
4. Pero `SessionProvider` no notifica a `useSession()` inmediatamente
5. El guard se ejecuta antes de que `useSession()` refleje la nueva sesión
6. `sessionStatus === 'unauthenticated'` (no `'loading'`) → el guard de loading no protege

### Qué intentamos (y falló)

| Intento | Commit | Resultado |
|---------|--------|-----------|
| `reconnectOnMount={false}` | `4e24d6f` → `ed853c9` | ❌ Rompió `useAccount().address` tras navegación |

## Solución propuesta (refinada)

### Opción A: Aflojar guard en páginas de solo lectura (recomendada — R-#167)

Modificar el guard "Partial login" en 3 páginas de lectura para permitir `address` sin `session`:

```tsx
// Antes (bloquea address && !session):
if ((session && !address) || (address && !session) || 
    (address && session && session.address && 
     address.toLowerCase() !== session.address.toLowerCase()))

// Después (solo bloquea sin wallet o con mismatch):
if (!address || (session && session.address && 
    address.toLowerCase() !== session.address.toLowerCase()))
```

**Archivos:** `app/[lang]/page.tsx`, `app/[lang]/[pathPrefix]/page.tsx`, `app/[lang]/[pathPrefix]/[pathSuffix]/page.tsx`

**No tocar:** `app/[lang]/profile/page.tsx` — necesita session para escrituras vía `authenticateUser()`.

**¿Por qué es seguro?**
- El modelo two-layer auth ([SIWE Auth Flow](../../doc/siwe-auth-flow.md)) protege las operaciones de escritura en la capa API, no en la UI
- `wagmi` conectado + `address` poblado = sabemos quién es el usuario
- La sesión se resolverá sola cuando NextAuth complete su poll
- Si se necesita el token para una operación, `getCsrfToken()` lo obtendrá

### Opción B: Forzar `session.update()` en navegación (alternativa futuro)

Agregar un `useEffect` que llame `update()` cuando `usePathname()` cambie:

```tsx
// En el layout o en un wrapper
const pathname = usePathname()
const { update } = useSession()

useEffect(() => {
  update() // Fuerza re-fetch de sesión en cada navegación
}, [pathname])
```

**Ventaja:** Resuelve la causa raíz sin cambiar guards. **Desventaja:** Patrón no documentado oficialmente, puede tener efectos secundarios con SIWE.

### Opción C: Custom SessionProvider con re-fetch (futuro lejano)

Implementar un wrapper de `SessionProvider` que re-fetchee en `pathname` change. [Patrón de la comunidad](https://github.com/nextauthjs/next-auth/discussions/5719). Requiere refactor mayor.
