# R-#152: Profiles of Church / GD Cluster

Build the foundation for Global Disciples cluster formation by enabling users to declare their church/place of worship, register as pastors, and form clusters of 3+ churches.

## Dependencies
None (base issue)

---

## MVP Checklist (Phase 1)

### 1. Church Relationship Declaration (User Profile)

Add a field to the user profile: **Church relationship** (`church_relationship`):

| Value | Description |
|-------|-------------|
| `pastor` | Pastor of a church |
| `leader` | Leader/elder |
| `member` | Member |
| *(empty)* | No relationship declared |

Only users with `church_relationship = 'pastor'` can declare a church and create/join clusters.

### 3. Location Data for Non-Colombia/Venezuela Countries

For countries outside Colombia and Venezuela (where `msip_departamento`, `msip_municipio`, `msip_centropoblado` are not pre-populated):

| Step | Action | Notes |
|------|--------|-------|
| **1** | User enters a city/town/village name in `place_of_worship_location` | Free text input with OpenCage autocomplete |
| **2** | Backend queries OpenCage Geocoder API | Uses the user's input + country to fetch structured data |
| **3** | OpenCage returns `components.state`, `components.county`, `components.city`, `components.town`, `components.village` | These map to department, municipality, city |
| **4** | If OpenCage returns valid data, create/upsert records in `msip_departamento`, `msip_municipio`, `msip_centropoblado` | Only for the specific location the user entered |
| **5** | If OpenCage returns no data or errors, store the raw text in `place_of_worship_location` | For MVP, allow user to enter text freely |

**Notes:**
- OpenCage API key is required. The free tier allows 2,500 requests/day.
- Cache results in the database to avoid repeated API calls for the same location.
- For MVP, do not block the user if OpenCage fails — allow manual entry.
- The API key only converts text to coordinates, but OpenCage's `components` field returns structured address data.

### 4.1 Data Privacy & Visibility

| Data | Visibility | Notes |
|------|-----------|-------|
| Church name, city, pastor name | **Public** | Displayed on church profile page |
| Church address | **Public** | Displayed on church profile page |
| ID photos (`id_photo_front`, `id_photo_back`) | **Private** | Used only for identity verification by pdJ admin. A "Delete" button removes them from the server. |
| Registration documents (`registration_photo`) | **Private** | Used only for verification by pdJ admin. Deletable via "Delete" button. |
| WhatsApp, Telegram, email | **Private** | Never sold or shared. Used for: (a) verification of profile info, (b) occasional platform announcements |
| Cluster data (name, members, funds) | **Public** | Visible on cluster page |

**Deletion rules:**
- When ID documents or registration files are deleted, **profile score is recalculated** (points from verified fields are lost).
- When profile data changes (name, location, church), verified fields are reset and points are lost until re-verified.

### 5. Profile Score (Updated)

| Component | Points | Condition |
|-----------|--------|-----------|
| **Name + country** | 68 | `country_id IS NOT NULL` |
| **WhatsApp or Telegram verified** | 12 | `verified_whatsapp = whatsapp` **OR** `verified_telegram = telegram` (12 max, not 24) |
| **Location (dept + municipality + city)** | 10 | `department_id IS NOT NULL AND verified_department_id = department_id AND verified_municipality_id = municipality_id AND verified_city_id = city_id` |
| **Place of worship** | 10 | `verified_place_of_worship = place_of_worship` |
| **Total** | **100** | Maximum |

**Notes:**
- Points are **not awarded automatically**. The user must click the "Update profile score" button on their profile page. This recalculates `profilescore` based on the current `verified_*` fields set by pdJ admin.
- Verification for Christians requires manual confirmation (WhatsApp/Telegram contact with the pastor).
- For non-Christians, location and place of worship are verified automatically upon completion.
- ID photos (Sierra Leone) and `registration_verified` do NOT give points; they are for KYC and legitimacy.
- The **church declaration** (§6) contributes to the "Place of worship" component: once `verified_place_of_worship = place_of_worship`, the user gains 10 points. There is no separate +32 reward — the church declaration's value is embedded in the Place of worship score.

### 7. Cluster Formation (Code-Based)

| Step | Description |
|------|-------------|
| **1. Create cluster** | Pastor creates a cluster from their church page → receives a 6-char alphanumeric code (e.g. `A3K9M2`) |
| **2. Share code** | Pastor shares the code with other pastors via WhatsApp, email, etc. |
| **3. Join cluster** | Other pastors enter the code on their church page → join the cluster |
| **4. Cluster formed** | When 3+ churches have joined, the cluster status changes to "Formed" |

**Rules:**
- A church can only belong to one cluster at a time. Joining a new cluster removes it from the previous one.
- No expiration on codes.
- No maximum member limit.
- Members can be added at any time after formation.
- All churches in a cluster must be from the same country.
- Cluster name must be unique within the same country (3-50 characters).
- Any pastor in the cluster can change the cluster name. Changes are logged in cluster history.
- Any pastor can click "Leave cluster" from their church page. Their church is removed from the cluster.
- When the last pastor leaves, the cluster is automatically dissolved.

### 8. Cluster Dissolution — Fund Handling

When a cluster dissolves (last pastor leaves), all funds held by the cluster are **transferred to the country fund** of the cluster's country. This logic is implemented in the `ClusterFunds` contract — see R-#155 §4.5 for the full specification.

| Scenario | Action |
|----------|--------|
| Pastor leaves and cluster becomes empty | Transfer all funds from cluster fund to country fund |
| Cluster is manually dissolved by admin | Transfer all funds from cluster fund to country fund |
| Country has no fund yet | Create country fund and transfer funds |

## Privacy Policy Compliance

Revisado contra `app/[lang]/privacy-policy/page.tsx`:

### Problemas encontrados

| # | Problema | Impacto |
|---|----------|---------|
| 1 | **item2** dice "el resto de la información en tu perfil es privada", pero las páginas de iglesia/clúster muestran públicamente `pastor_name`, `city_name` y `church.name` | Contradicción: datos declarados privados se muestran públicos |

| Prioridad | Acción |
|-----------|--------|
