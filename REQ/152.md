# R-#152: Church Profiles, Pastors, and Places of Worship

Enable users to declare their church/place of worship, register as pastors, and manage their profile with town-based location data and verification scoring.

## Dependencies
None (base issue)

---

## MVP Checklist (Phase 1)

### 1. Church Relationship Declaration (User Profile)

Add a field to the user profile: **Church relationship** (`church_relationship`):

| Value | Description |
|-------|-------------|
| `pastor` | Pastor of a church |
| `leader` | Leader/elder |
| `member` | Member |
| *(empty)* | No relationship declared |

Anyone can declare a new church via "+ New church" in the selector with fields: church name, pastor name, pastor WhatsApp.

The first person who declares themselves as pastor of that church **and** has an interview date (`date_of_interview IS NOT NULL`) gains access to the church dialog (see §7) which additionally includes registration number and registration photo.

If the pastor clears their interview date (e.g., via "Delete Verified Data"), their `church_relationship` reverts to `member` and they lose access to edit the church data. Another user can then claim the pastor role by setting their relationship to `pastor` and scheduling an interview.

### 3. Location Data (Implemented)

Location selection uses DB-driven town autocomplete via `/api/towns/search`. The user types a town name and gets suggestions from `msip_centropoblado` (joined with `msip_municipio` and `msip_departamento`) in the format "Town, Municipality, Department". Selecting a suggestion sets `city_id`. Free text is stored as `place_of_worship_location` with `city_id = null`.

| Step | Action | Status |
|------|--------|--------|
| **1** | User types town name → autocomplete via `/api/towns/search` (DB-driven, no third party) | ✅ Done |
| **2** | User selects suggestion → `city_id` set from `msip_centropoblado` | ✅ Done |
| **3** | User types free text → stored in `place_of_worship_location`, `city_id = null` | ✅ Done |
| **4** | Church list filtered by selected town (`cityId`) | ✅ Done |

OpenCage removed in favor of local DB data (Colombia, Venezuela, Honduras, Sierra Leone have full coverage). For countries without DB data, free text input is used.

### 4. Church Table (`church`)

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `name` | VARCHAR(255) | ✅ Yes | |
| `country_id` | INTEGER | ✅ Yes | |
| `department_id` | INTEGER | ❌ No | |
| `municipality_id` | INTEGER | ❌ No | |
| `city_id` | INTEGER | ❌ No | References `msip_centropoblado` |
| `city` | TEXT | ❌ No | Mirrors `place_of_worship_location` of the user who declared the church. Used when the town is not in our DB (e.g., "Seria, Falaba"). |
| `address` | TEXT | ❌ No | |
| `pastor_name` | VARCHAR(100) | ✅ Yes | |
| `pastor_whatsapp` | VARCHAR(20) | ✅ Yes | |
| `pastor_telegram` | VARCHAR(50) | ❌ No | |
| `pastor_id` | INTEGER | ❌ No | References `usuario`. Set when a user with `church_relationship = 'pastor'` AND `date_of_interview IS NOT NULL` claims this church. |
| `denomination` | VARCHAR(100) | ❌ No (Phase 2) | |
| `registration` | VARCHAR(50) | ✅ Yes (for pastors) | Required when a pastor creates or claims a church. **Only visible in the pastor dialogue.** |
| `registration_photo` | TEXT | ✅ Yes (for pastors) | File path to photo of registration document. **Only visible in the pastor dialogue.** |
| `registration_verified` | BOOLEAN | ❌ No | Set by pdJ admin after verifying the registration document |
| `created_by` | INTEGER | ✅ Yes | References `usuario` (who created the church record) |

**Notes:**
- Anyone can declare a new church with name, pastor name, pastor WhatsApp. The church's `city` is set from the user's `place_of_worship_location`.
- `registration` and `registration_photo` are only visible in the pastor dialog (see §7), accessible only to the confirmed pastor (user with `church_relationship = 'pastor'` **and** `date_of_interview IS NOT NULL`).
- If the pastor clears their interview date, they revert to `member` and lose access to the pastor dialog.
- `registration_verified` does NOT give profile score points, but provides assurance that the pastor is legitimate.
- The `registration_photo` is stored in the local file system and **served via API** (not `public/`) for confidentiality — see privacy policy `en/privacy-policy`.
- `id_photo_front` and `id_photo_back` (Sierra Leone only) follow the same pattern: stored in file system, served via API.

### 4.1 Data Privacy & Visibility

| Data | Visibility | Notes |
|------|-----------|-------|
| Church name, city, pastor name | **Public** | Displayed on church profile page |
| Church address | **Public** | Displayed on church profile page |
| ID photos (`id_photo_front`, `id_photo_back`) | **Private** | Used only for identity verification by pdJ admin. A "Delete" button removes them from the server. |
| Registration documents (`registration_photo`) | **Private** | Used only for verification by pdJ admin. Deletable via "Delete" button. |
| WhatsApp, Telegram, email | **Private** | Never sold or shared. Used for: (a) verification of profile info, (b) occasional platform announcements |

**Deletion rules:**
- When ID documents or registration files are deleted, **profile score is recalculated** (points from verified fields are lost).
- When profile data changes (name, location, church), verified fields are reset and points are lost until re-verified.

### 5. Profile Score (Updated)

Calculated by `recalculateProfileScore()` in `lib/gd-utils.ts`, called from:
- `POST /api/update-scores` ("Update Scores" button)
- `PATCH /api/profile` (after "Save Changes")

| Component | Points | Condition |
|-----------|--------|-----------|
| **Name verified** | 25 | `nombre = passport_name` |
| **Country verified** | 25 | `pais_id = passport_nationality` |
| **Email verified** | 9 | `email = verified_email` |
| **WhatsApp or Telegram verified** | 9 | `verified_whatsapp = whatsapp` **OR** `verified_telegram = telegram` (9 max, not 18) |
| **GoodDollar verified** | 7 | `lastgooddollarverification IS NOT NULL` |
| **Location verified** | 9 | `department_id IS NOT NULL AND verified_department_id = department_id AND verified_municipality_id = municipality_id AND verified_city_id = city_id` |
| **Place of worship verified** | 9 | `verified_place_of_worship = place_of_worship` |
| **Interview with verifier** | 7 | `date_of_interview IS NOT NULL` |
| **Total** | **100** | |

**Notes:**
- Points are **not awarded automatically**. The user must click "Update Scores" or "Save Changes" (which also recalculates).
- Verification for Christians requires manual confirmation (WhatsApp/Telegram contact with the pastor).
- ID photos (Sierra Leone) and `registration_verified` do NOT give points; they are for KYC and legitimacy.
- Verified pastors in Sierra Leone and Colombia receive a **44 SLEARN bonus** after full verification (see R-#190).

### 7. Pastor Dialogue (New Church)

When a Christian user with `church_relationship = 'pastor'` selects "+ New church" from the church selector, a dialogue opens with these additional fields:

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| Church name | Text | ✅ Yes | Pre-filled from the selector's text input |
| Pastor name | Text | ✅ Yes | |
| Pastor WhatsApp | Text | ✅ Yes | |
| Registration number | Text | ✅ Yes | Registration number |
| Registration photo | File upload | ✅ Yes | Photo of registration document |

**Rules:**
- Anyone can create a new church via "+ New church". Basic fields: church name, pastor name, pastor WhatsApp.
- The church dialog (with registration number + registration photo) is only accessible to the confirmed pastor: a user whose `church_relationship = 'pastor'` **and** `date_of_interview IS NOT NULL`.
- If the pastor clears their interview date (e.g., "Delete Verified Data"), they revert to `member` and lose access to the church dialog. Another user can claim the pastor role.
- Non-Christians skip the church selector entirely — they only fill `place_of_worship` as free text.
