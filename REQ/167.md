# R-#167: Fix Session Persistence Across Page Navigation (Partial Login)

Después de conectar wallet (OneKey) en `https://learn.tg/`, firmar SIWE, y navegar a `/en`, el usuario ve: _"Partial login. Please disconnect your wallet and connect and sign again."_ El problema se repite en navegación entre cursos, guías y landing pages, degradando la UX especialmente en móvil y PWA en Sierra Leona.

**Issues fusionados:** R-#167 + R-#173 (misma causa raíz, mismo fix).

## Síntoma

1. Abrir `https://learn.tg` en nueva pestaña
2. Activar OneKey → recargar
3. Conectar wallet → confirmar → firmar SIWE
4. Clic en English → `/en` muestra "Partial login"

El test E2E con Puppeteer + SIWE simulado (R-#172) **no reproduce** el bug — solo ocurre con wallet real (OneKey). Ver [R-#172](172.md) para detalles del test.

## Causa Raíz

**NextAuth bug conocido:** `useSession()` retorna `status: 'unauthenticated'` tras navegación cliente (`<Link>`, `router.push()`), incluso con cookie `next-auth.session-token` válida.

- [NextAuth discussion #5719](https://github.com/nextauthjs/next-auth/discussions/5719) — 37 participantes
- [NextAuth issue #9504](https://github.com/nextauthjs/next-auth/issues/9504)

**Mecanismo:** El `SessionProvider` no re-fetchea la sesión automáticamente en navegación cliente. Su polling interno (`refetchInterval`) solo dispara periódicamente, no en `router.push()`. Cuando el componente se monta en la nueva ruta, `useSession()` recibe el estado inicial del contexto (`null`) antes de que el poll se ejecute.

**Por qué el E2E no lo reproduce:** El test simula SIWE vía POST directo y verifica la cookie, pero no monta `SessionProvider` ni ejecuta `useSession()` en cliente React.

**Por qué con OneKey sí ocurre:**
1. Wagmi se reconecta en mount → `address` poblado
2. `RainbowKitSiweNextAuthProvider` detecta wallet → dispara SIWE
3. SIWE completa → cookie seteada
4. Pero `SessionProvider` no notifica a `useSession()` inmediatamente
5. El guard se ejecuta antes de que `useSession()` refleje la nueva sesión
6. `sessionStatus === 'unauthenticated'` (no `'loading'`) → el guard de loading no protege

### Lo que aprendimos

1. **`reconnectOnMount={false}` no es viable solo** — wagmi pierde la dirección tras navegación y el guard bloquea todo
2. **La causa raíz no es wagmi, es NextAuth** — `useSession()` retorna `'unauthenticated'` tras navegación cliente incluso con cookie válida
3. **El modelo two-layer auth de learn.tg da flexibilidad** — capa UI (session) y capa API (token DB) están desacopladas; páginas de solo lectura no necesitan session

## Solución

**Aflojar el guard "Partial login" en páginas de solo lectura** (courses, guides, landing):

```tsx
// Actual (3 condiciones, muy estricto):
if ((session && !address) || (address && !session) || 
    (address && session && session.address && 
     address.toLowerCase() !== session.address.toLowerCase()))

// Propuesto (2 condiciones, permite address sin session):
if (address && session && session.address && 
    address.toLowerCase() !== session.address.toLowerCase())
```

**Razonamiento:**
- Si hay wallet conectada (`address`) y sesión coincidente → renderiza normalmente
- Sin wallet (`!address`) → la página muestra contenido público sin datos de usuario
- Wallet conectada sin sesión (`address && !session`) → caso NextAuth bug, renderiza y la sesión se resuelve sola
- Las operaciones de escritura se protegen en la capa API vía `authenticateUser()` → `billetera_usuario.token`
- Solo bloqueamos si AMBAS direcciones existen y NO coinciden (mismatch real)

**Archivos modificados:**
- `app/[lang]/page.tsx` — landing de cursos (useEffect guard + render guard)
- `app/[lang]/[pathPrefix]/page.tsx` — página de curso (render guard)
- `app/[lang]/[pathPrefix]/[pathSuffix]/page.tsx` — página de guía (render guard)
- `lib/hooks/useCourse.ts` — hook de fetch de curso (data-fetching guard)
- `app/[lang]/[pathPrefix]/[pathSuffix]/test/page.tsx` — crucigrama (guard + carga)

**No modificar:** `app/[lang]/profile/page.tsx` — el perfil necesita el token de `billetera_usuario` para escrituras vía `authenticateUser()`.

### Detalle del fix en `test/page.tsx` (crucigrama)

**Problema:** Si NextAuth retorna `'unauthenticated'` tras navegación (bug #5719) pero wagmi tiene `address`:
1. La carga del crucigrama se saltaba (`!session` en línea 110) → grid vacío
2. El guard mostraba "Connect Wallet" aunque la wallet SÍ estaba conectada

**Fix:**
1. `!session` removido de la condición de carga — el crucigrama carga con `getCsrfToken()` que funciona sin sesión autenticada
2. Guard aflojado: solo bloquea si `!address` (sin wallet) o mismatch de direcciones
3. La seguridad en submit se mantiene: `/api/check-crossword` usa `authenticateUser()` que valida `walletAddress + token` contra `billetera_usuario`

## Alternativas Consideradas (futuro)

| Alternativa | Viabilidad | Nota |
|-------------|-----------|------|
| `SessionUpdater` con `update()` en navegación | Alta | `useEffect(() => update(), [pathname])` — ataca causa raíz, pero añade requests por navegación |
| Custom `SessionProvider` que re-fetchea en `usePathname()` | Media | Resuelve raíz pero añade complejidad — [patrón comunidad](https://github.com/nextauthjs/next-auth/discussions/5719) |
| `RainbowKitAuthenticationProvider` con status manual | Media | Rompe acoplamiento wallet↔SIWE |
| Pasar session desde server component | Baja | Requiere refactor grande |
| NextAuth v5 con `auth()` | Baja | Requiere migración de versión |

### Descartado
- Defer SIWE verification — RainbowKit lo maneja automáticamente
- localStorage session cache — riesgo XSS, duplica NextAuth
