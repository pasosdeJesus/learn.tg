# R-#184: Wallet managed by the user without server

Create a PWA-native wallet that lets new users generate a self-custodied wallet
directly in the browser — no app store, no extension, no server storage. The
user receives a seed phrase and controls their keys from day one.

## Motivation

Many users arrive at learn.tg **without a wallet**. Today they must:
1. Leave learn.tg → go to Play Store / App Store
2. Install OneKey (or another wallet)
3. Return to learn.tg, connect, and sign in

This friction loses users. A built-in wallet eliminates the bounce and lets
users start learning immediately. It also aligns with our educational mission:
the user learns seed phrases by **using** one from the start.

## Design

### Wallet Creation (Client-Side Only)

```
┌─────────────────────────────────────────────────────┐
│  Browser / PWA                                       │
│                                                      │
│  1. crypto.getRandomValues() → 128-bit entropy       │
│  2. BIP39 → 12-word seed phrase                      │
│  3. BIP32/BIP44 → private key → address              │
│  4. Show seed phrase to user (once, never again)      │
│  5. User confirms they saved it                      │
│  6. Sign SIWE message with derived private key → auth │
│                                                      │
│  Nothing leaves the browser — no server knows         │
│  the seed phrase or private key.                      │
└─────────────────────────────────────────────────────┘
```

### Storage

| What | Where | Encrypted? |
|------|-------|------------|
| Seed phrase | User's paper / device | N/A |
| Private key | Session-only memory (or IndexedDB encrypted with user PIN) | Optional PIN |
| Public address | `billetera_usuario` (same as any wallet) | N/A |

No server-side storage of private material. The private key can optionally be
stored in IndexedDB encrypted with a user-chosen PIN for session persistence
(avoids re-entering seed phrase every visit).

### Auth Flow

The derived private key signs SIWE messages exactly like an external wallet.
No change to `/api/auth/callback/credentials` — the backend sees a valid SIWE
signature and proceeds normally.

### Bridge to OneKey

Users who later install OneKey can import their seed phrase — the same address
and funds appear in both the PWA wallet and OneKey. This makes the PWA wallet
a **stepping stone**, not a dead end.

## Dependencies

- `viem` or `ethers` (already dependencies)
- `bip39` + `@scure/bip32` (lightweight, ~15KB gzipped)
- Web Crypto API (`crypto.getRandomValues()`) — available in all modern browsers and PWAs

## Out of Scope

- Native mobile app (PWA only)
- Storing seed phrase server-side
- Sending transactions from the PWA wallet (SIWE auth only — user switches to
  OneKey for UBI claims and scholarships)
- Multi-chain (Celo only)

## MVP Scope

| Component | Notes |
|-----------|-------|
| **Wallet generator** | Client-side: entropy → seed phrase → private key → address |
| **SIWE signer** | Sign SIWE messages with derived key (no `eth_requestAccounts`) |
| **Seed phrase UI** | Show 12 words, confirm user wrote them down, never show again |
| **Optional PIN encryption** | Encrypt private key in IndexedDB with user PIN for persistence |
| **OneKey migration guide** | Link in profile/docs explaining how to import seed into OneKey |

## Acceptance Criteria

- [ ] User clicks "Create Wallet" → sees 12-word seed phrase
- [ ] User confirms phrase → wallet is "connected" (address derived, no server roundtrip)
- [ ] User can sign in (SIWE) with the derived key
- [ ] Private material never leaves the browser (verify via Network tab)
- [ ] Works in PWA standalone mode
- [ ] Seed phrase can be imported into OneKey and yields same address
