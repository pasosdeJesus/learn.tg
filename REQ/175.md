# R-#175: E2E Test — Claim CELO UBI + GoodDollar from Guide Page

> *"El que es fiel en lo muy poco, también en lo más es fiel"* (Lucas 16:10)

---

## Summary

Crear un test E2E con Puppeteer + SIWE simulado que navegue a la guía del curso
Web3 & UBI, autentique al usuario, y reclame **Learn.tg-UBI (CELO)** y
**GoodDollar UBI** desde los botones embebidos en la misma guía.

---

## Background

La guía del curso Web3 & UBI (`/en/web3-and-ubi/guide3` en producción) contiene dos
marcadores mágicos en su markdown:

| Marcador | Componente | API llamada |
|----------|-----------|-------------|
| `{CeloUbiButton}` | `CeloUbiButton.tsx` | `POST /api/claim-celo-ubi` |
| `{GoodDollarButton}` | `GoodDollarClaimButton.tsx` | `claimSDK.claim()` + `POST /api/register-gooddollar-claim` |

Actualmente no hay cobertura E2E de estos flujos críticos. Ambos botones usan
`session.address` + `getCsrfToken()` para autenticación, siguiendo el modelo
two-layer auth documentado en `doc/siwe-auth-flow.md`.

## Component Analysis

### CeloUbiButton — ✅ Testeable con Puppeteer + SIWE simulado

```tsx
// Flujo simplificado:
async handleClaimClick() {
  const csrfToken = await getCsrfToken()
  const response = await axios.post('/api/claim-celo-ubi', {
    walletAddress: session.address,
    token: csrfToken,
  })
  // Muestra diálogo con resultado (success/error)
}
```

- **Requiere:** `session.address` presente + `getCsrfToken()` funcional
- **Mockeable:** Sí — `simulateSIWE()` ya produce una sesión válida con cookie
- **Selector del botón:** Texto "Claim Learn.tg-UBI" o "Reclamar Learn.tg-IBU"
- **Selector de éxito:** Diálogo con "Claim Successful" / "Reclamo Exitoso"
- **Verificable:** Texto del diálogo, presencia de txHash

### GoodDollarClaimButton — ⚠️ Parcialmente testeable

```tsx
async handleClaim() {
  // Requiere: session, address, publicClient, walletClient, identitySDK
  const claimSDK = new ClaimSDK({ account, publicClient, walletClient, identitySDK, env })
  const result = await claimSDK.claim()                    // ← requiere wallet real
  await fetch('/api/register-gooddollar-claim', { ... })   // ← registra en backend
}
```

- **Requiere:** `useAccount().address` (wagmi), `usePublicClient()` (wagmi),
  `useWalletClient()` (wagmi), `useIdentitySDK()` (GoodDollar SDK)
- **Mockeable:** ❌ No con Puppeteer puro — wagmi requiere un provider EIP-6963
  real. El mock actual de `window.ethereum` en `helpers.mjs` no es detectado
  por wagmi como conector.
- **Selector del botón:** "Sign up with GoodDollar or Claim UBI" /
  "Regístrate con GoodDollar o reclama UBI"
- **Selector de estado sin wallet:** "Connect your wallet to claim" /
  "Conecta tu billetera para reclamar"

## Proposed Solution

### Test 1: CeloUbiButton — Claim flow completo ✅

```
1. launchBrowser() + newPage() con mock de ethereum
2. goto(`${base}/en/web3-and-ubi/guide3`)
3. simulateSIWE() → sesión válida con cookie
4. Esperar que el botón "Claim Learn.tg-UBI" aparezca
5. Click en el botón
6. Esperar diálogo con "Claim Successful" o "Reclamo Exitoso"
7. Verificar que el diálogo contiene txHash o amount
8. Cerrar diálogo
9. **Devolver el CELO reclamado** al backend wallet (`NEXT_PUBLIC_ADDRESS`)
   para mantener el fondo UBI circular
```

### Mecanismo de devolución de CELO

Después del claim exitoso, el test transfiere automáticamente el CELO recibido
de vuelta a la dirección del backend que fondea el contrato CeloUbi. Esto
mantiene el fondo UBI circular y evita que los tests drenen los fondos reales.

```js
import { createWalletClient, http, parseEther } from 'viem'
import { celo } from 'viem/chains'

async function returnCeloToBackend(account, backendAddress) {
  const client = createWalletClient({
    account,
    chain: celo,
    transport: http(),
  })
  const balance = await client.getBalance({ address: account.address })
  if (balance === 0n) return null
  const gas = parseEther('0.0001')
  const amount = balance - gas
  const hash = await client.sendTransaction({
    to: backendAddress,
    value: amount,
  })
  return hash
}
```

La dirección de backend se lee de `NEXT_PUBLIC_ADDRESS` en `apps/.env`.
El txHash de devolución se registra en el output del test para trazabilidad.

**Verificaciones:**
- Botón visible después de SIWE
- Diálogo de éxito aparece tras click
- No hay mensaje "You must be logged in"

### Test 2: GoodDollarClaimButton — Render condicional ⚠️

```
1. Nueva página (misma sesión SIWE del Test 1 o fresh)
2. goto(`${base}/en/{coursePrefix}/celo-ubi`)
3. Verificar que el botón "Sign up with GoodDollar or Claim UBI" renderiza
4. Verificar que el mensaje "Connect your wallet to claim" aparece
     (porque wagmi no tiene wallet real conectada en Puppeteer)
```

**Verificaciones:**
- Botón de GoodDollar se renderiza en la guía
- Mensaje de "Connect your wallet" visible (esperado sin provider wagmi)
- El botón NO crashea la página (regresión R-#168 — QR crash)

**Limitación documentada:** El claim real de GoodDollar requiere `walletClient`
de wagmi, que solo funciona con un provider EIP-6963 inyectado por una
extensión real (OneKey, MetaMask). Testing del claim completo requeriría:
- Extensión de wallet cargada en Puppeteer (`--load-extension`)
- O un mock EIP-6963 compatible con wagmi (REQ futuro — ver R-#172 §"Long-Term Goal")
- O test manual con `CONCABEZA=1` y wallet real

### Test 3: Ambos botones en misma página

```
1. goto(`${base}/en/{coursePrefix}/celo-ubi`)
2. simulateSIWE()
3. Verificar que AMBOS botones coexisten en la página
4. Verificar orden: CeloUbiButton antes que GoodDollarClaimButton
     (según orden en page.tsx:237-238)
```

### Determinar el coursePrefix real

El test debe detectar dinámicamente el curso que contiene la guía con ambos botones.
Alternativas:

1. **Hardcodear:** El curso es fijo y conocido: `/en/web3-and-ubi/guide3`.

**Default:** `/en/web3-and-ubi/guide3`. La variable `GUIDE_CLAIM_PATH` permite
sobrescribir para staging o entornos locales.

## Pseudocódigo del test

```js
// guide-claims.mjs
import {
  initTestEnv, launchBrowser, newPage,
  resetFailures, fail, ok, summary,
  simulateSIWE, waitForText,
  short,
} from './helpers.mjs'

async function main() {
  resetFailures()
  const env = initTestEnv()
  const { base, timeout, account, chainId, host, domainPort } = env
  const guidePath = process.env.GUIDE_CLAIM_PATH || '/en/web3-and-ubi/guide3'

  console.log(`🧪 Guide claims E2E — ${guidePath}\n`)
  const browser = await launchBrowser(env.headless)

  // ── Test 1: CeloUbiButton claim ──────────────────────────
  {
    const page = await newPage(browser, account.address, timeout)
    await page.goto(`${base}${guidePath}`, { waitUntil: 'domcontentloaded', timeout })
    
    const siweOk = await simulateSIWE(page, { account, host, domainPort, base, chainId })
    if (!siweOk) { fail('SIWE falló'); await browser.close(); process.exit(1) }
    ok('SIWE completado')

    // Esperar botón y hacer click
    await page.waitForFunction(() =>
      document.body?.textContent?.includes('Claim Learn.tg-UBI') ||
      document.body?.textContent?.includes('Reclamar Learn.tg-IBU'),
      { timeout: 15000 })
    ok('Botón CeloUbi visible')

    const btn = await page.evaluateHandle(() =>
      [...document.querySelectorAll('button')].find(b =>
        b.textContent?.includes('Claim Learn.tg-UBI') ||
        b.textContent?.includes('Reclamar Learn.tg-IBU')))
    await btn.asElement().click()
    
    const success = await waitForText(page, 'Claim Successful', 30) ||
                    await waitForText(page, 'Reclamo Exitoso', 5)
    if (success) ok('Diálogo de éxito CeloUbi visible')
    else fail('Diálogo de éxito CeloUbi no apareció')

    await page.close()
  }

  // ── Test 2: GoodDollarClaimButton render ─────────────────
  {
    const page = await newPage(browser, account.address, timeout)
    await page.goto(`${base}${guidePath}`, { waitUntil: 'domcontentloaded', timeout })
    await simulateSIWE(page, { account, host, domainPort, base, chainId })

    const gdVisible = await waitForText(page, 'Sign up with GoodDollar', 15) ||
                       await waitForText(page, 'Regístrate con GoodDollar', 5)
    if (gdVisible) ok('Botón GoodDollar visible')
    else fail('Botón GoodDollar no visible')

    const connectMsg = await page.evaluate(() =>
      document.body?.textContent?.includes('Connect your wallet') ||
      document.body?.textContent?.includes('Conecta tu billetera'))
    if (connectMsg) ok('Mensaje "Connect wallet" visible (esperado sin EIP-6963)')
    else console.log('  ⚠️  Mensaje "Connect wallet" no visible')

    await page.close()
  }

  // ── Test 3: Coexistencia de botones ──────────────────────
  {
    const page = await newPage(browser, account.address, timeout)
    await page.goto(`${base}${guidePath}`, { waitUntil: 'domcontentloaded', timeout })
    await simulateSIWE(page, { account, host, domainPort, base, chainId })
    await new Promise(r => setTimeout(r, 3000))

    const buttons = await page.evaluate(() =>
      [...document.querySelectorAll('button')].map(b => b.textContent?.trim()))
    const hasCeloUbi = buttons.some(t => t?.includes('Claim Learn.tg-UBI') || t?.includes('Reclamar Learn.tg-IBU'))
    const hasGoodDollar = buttons.some(t => t?.includes('Sign up with GoodDollar') || t?.includes('Regístrate con GoodDollar'))

    if (hasCeloUbi && hasGoodDollar) ok('Ambos botones coexisten en la guía')
    else fail(`Falta uno: CeloUbi=${hasCeloUbi}, GoodDollar=${hasGoodDollar}`)

    await page.close()
  }

  await browser.close()
  const failures = summary(performance.now())
  process.exit(failures > 0 ? 1 : 0)
}

main().catch(err => { console.error('❌', err.message); process.exit(1) })
```
