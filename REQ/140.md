# Requirement: 

**ID**: R-#XXX
**Title**: Implement Passkey (WebAuthn) authentication as the primary daily login method, maintaining wallet signature for initial linking and user-initiated financial operations.
**Date**: May 2026

---

## Objective

Replace SIWE as the daily authentication method in learn.tg with **Passkeys (WebAuthn)**, eliminating the friction of wallet reconnection when the session expires. The wallet will still be required for:

1. **Linking to a passkey** (one-time, with a signature that includes the passkey).
2. **User-initiated financial operations** (donations to course vaults, premium course payments).

---

## Motivation

- **SIWE expires and creates friction**: When the JWT expires, the user must sign again with their wallet. On basic devices with limited connectivity (common in Sierra Leone), this can take 30 seconds to 2 minutes and cause the PWA to lose its state.
- **SIWE conflicts with PWA**: In standalone mode, switching from the PWA to the wallet and back is a poor experience.
- **The backend already signs most transactions**: Scholarships, UBI, and future SBTs are signed by the backend (`PRIVATE_KEY`). The user does not need to sign on-chain to receive rewards.
- **Passkeys are the modern standard**: Supported by all browsers, they work without special hardware (the device PIN is sufficient), and are resistant to phishing.

---

## Design Overview

```
REGISTRATION (one-time per device):
  1. User connects wallet (any WalletConnect-compatible wallet)
  2. User registers a passkey on the device
  3. User signs with wallet a message that includes the passkey public key
  4. Backend verifies the signature and stores the link: passkey_id ↔ wallet_address

KYC (independent operation, one-time):
  - User verifies identity with self.xyz (NFC + passport)
  - self.xyz returns: wallet_address, name, country (ZK proof)
  - learn.tg stores: user_id, name, country
  - Without KYC, the user can use learn.tg but CANNOT claim scholarships or UBI

DAILY OPERATIONS (with passkey):
  1. User authenticates with passkey (PIN/fingerprint/device pattern)
  2. Backend issues JWT with claims: { user_id, wallet_address, device_id }
  3. Operations using JWT (no wallet, no additional signature):
     - View courses and guides
     - View profile and progress
     - Save/edit profile
     - Submit crossword (backend signs submitGuideResult)
     - Claim UBI (backend signs claim)
     - Receive SBT (backend signs mint)

USER-INITIATED FINANCIAL OPERATIONS (require real wallet):
  - Donate to course vault → WalletConnect → user signs transferFrom
  - Pay for premium course → WalletConnect → user signs transfer
```

## Detailed Technical Flow

### 1. Passkey ↔ Wallet Linking (One-Time)

```
Prerequisites:
  - User has a wallet (OneKey, OKX, MetaMask, etc.)
  - Browser supports WebAuthn

Flow:
  1. User clicks "Link Passkey"
  2. Frontend calls navigator.credentials.create() to register a new passkey
     - The passkey is bound to the device and domain (learn.tg)
  3. Frontend extracts the passkey's public key (credentialPublicKey)
  4. Frontend sends to backend: { wallet_address, passkey_public_key }
  5. Backend generates a challenge message:
     "I am linking this passkey to my learn.tg account.
      Passkey public key: <base64url encoded>
      Timestamp: <ISO 8601>
      Wallet address: <0x...>"
  6. User signs this message with their wallet
  7. Frontend sends to backend: { wallet_address, passkey_public_key, signature, message }
  8. Backend verifies:
     a. The signature is valid and corresponds to wallet_address
     b. The signed message contains the correct passkey public key
     c. The timestamp is recent (within 5 minutes)
  9. If valid, backend stores:
     - passkey_credentials: { user_id, credential_id, public_key, device_name, created_at }
     - wallet_link: { user_id, wallet_address, passkey_id, verified_at }

Security considerations:
  - The signature MUST include the passkey public key to prevent linking a different passkey
  - The message MUST include a timestamp to prevent replay attacks
  - The backend MUST verify that the wallet_address in the signature matches the one provided
```

### 2. Daily Authentication with Passkey

```
Flow:
  1. User visits learn.tg (PWA or browser)
  2. User clicks "Sign in with Passkey"
  3. Frontend calls navigator.credentials.get() with the registered credential_id
  4. User verifies with device PIN, fingerprint, or pattern
  5. Browser returns an assertion (signed challenge)
  6. Frontend sends assertion to backend: POST /api/auth/passkey
  7. Backend verifies:
     a. The assertion is valid for the stored public key
     b. The challenge matches the one sent
     c. The credential_id is linked to a user
  8. Backend issues JWT with claims:
     {
       sub: user_id,
       wallet: wallet_address,
       device_id: credential_id,
       iat: timestamp,
       exp: timestamp + 24h
     }
  9. Frontend stores JWT and uses it for all API calls

JWT Renewal:
  - When JWT is close to expiring, frontend can request renewal
  - If passkey is still available on the device, backend issues a new JWT
  - No user interaction required (the passkey session is still active)
```

### 3. Self.xyz KYC (Independent Operation)

```
Flow:
  1. User clicks "Verify Identity"
  2. App redirects to self.xyz or opens its widget
  3. User scans passport with NFC
  4. self.xyz generates ZK proof: { wallet_address, name, country, age >= 18 }
  5. self.xyz returns this proof to learn.tg callback
  6. learn.tg backend receives: { wallet_address, name, country }
  7. Backend looks up user by wallet_address and updates:
     - user.verified_name = name
     - user.country = country
     - user.kyc_completed_at = now

Important:
  - self.xyz returns wallet_address, NOT user_id
  - The wallet_address is the link between self.xyz data and learn.tg user
  - KYC is NOT required for passkey ↔ wallet linking
  - KYC is NOT required for basic platform usage
  - KYC IS required for:
    - Claiming UBI (CeloUbi.claim)
    - Receiving scholarships (LearnTGVaults.submitGuideResult)
    - Participating in SLEARN distribution

Note:
  - Users without KYC can still:
    - View courses and guides
    - Complete crosswords (for learning, no rewards)
    - Link passkey and wallet
    - Earn Learning Points (if allowed by current rules)
```

### 4. Operations Authorization Matrix

```
┌──────────────────────────────────────────────────────────────────┐
│                        AUTHORIZATION MATRIX                       │
├────────────────────────────┬───────────┬──────────┬──────────────┤
│ OPERATION                  │ PASSKEY   │ WALLET   │ KYC REQUIRED │
│                            │ (JWT)     │ (SIGN)   │              │
├────────────────────────────┼───────────┼──────────┼──────────────┤
│ View courses/guides        │    ✅     │    ❌    │     ❌       │
│ View profile/progress      │    ✅     │    ❌    │     ❌       │
│ Save/edit profile          │    ✅     │    ❌    │     ❌       │
│ Submit crossword (learning)│    ✅     │    ❌    │     ❌       │
│ Submit crossword (reward)  │    ✅     │    ❌    │     ✅       │
│ Claim UBI                  │    ✅     │    ❌    │     ✅       │
│ Receive SBT                │    ✅     │    ❌    │     ❌*      │
│ Donate to course vault     │    ❌     │    ✅    │     ❌       │
│ Pay for premium course     │    ❌     │    ✅    │     ❌       │
│ KYC with self.xyz          │    ❌     │    ❌    │   (creates)  │
└────────────────────────────┴───────────┴──────────┴──────────────┘

* SBT receipt does not require KYC, because it's a proof of course completion,
  not a financial reward. However, the SBT's utility in stable-sl may require KYC.

For operations that REQUIRE WALLET SIGNATURE:
  - The user must connect wallet via WalletConnect
  - These operations are infrequent (once per donation/purchase)
  - The friction of using the wallet is acceptable given the financial nature
```

## Database Changes

### New Table: `passkey_credentials`

```sql
CREATE TABLE passkey_credentials (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    usuario_id INTEGER NOT NULL REFERENCES usuario(id),
    credential_id TEXT NOT NULL UNIQUE,       -- WebAuthn credential ID
    public_key BYTEA NOT NULL,                -- WebAuthn public key (COSE format)
    device_name TEXT,                         -- e.g., "Chrome on Android", "Safari on iPhone"
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    last_used_at TIMESTAMP,
    UNIQUE(usuario_id, credential_id)         -- One passkey per device per user
);
```

### New Table: `wallet_passkey_link`

```sql
CREATE TABLE wallet_passkey_link (
    id SERIAL PRIMARY KEY,
    usuario_id INTEGER NOT NULL REFERENCES usuario(id),
    wallet_address TEXT NOT NULL,             -- 0x... format, case insensitive
    passkey_id UUID NOT NULL REFERENCES passkey_credentials(id),
    verified_at TIMESTAMP NOT NULL DEFAULT NOW(),
    UNIQUE(usuario_id, wallet_address),
    UNIQUE(usuario_id, passkey_id)
);
```

### Modification to `billetera_usuario`

```sql
-- Add passkey_link_id column (optional, for backward compatibility)
ALTER TABLE billetera_usuario 
ADD COLUMN passkey_link_id UUID REFERENCES wallet_passkey_link(id);
```

## API Endpoints

|---|---|---|---|
| `POST` | `/api/auth/passkey/register/challenge` | None | Get challenge for passkey registration |
| `POST` | `/api/auth/passkey/register` | None | Complete passkey registration |

## Security Considerations

### Threats and Mitigations

| Threat | Mitigation |
|---|---|
| **JWT theft (XSS, malware)** | JWT bound to device_id claim; backend can detect unusual device changes |
| **Passkey linking to wrong wallet** | Wallet signature MUST include the passkey public key; backend verifies both |
| **Replay of passkey assertion** | Challenge-response with timestamp; each challenge is single-use |
| **User loses passkey device** | Allow multiple passkeys per user (phone + computer); provide recovery via wallet signature |
| **KYC evasion** | Backend checks KYC status before processing rewards; self.xyz proof is cryptographically verified |
| **Man-in-the-middle** | All communication over HTTPS; WebAuthn is origin-bound (phishing-resistant) |

### Recovery Flow

```
If user loses their passkey device:
  1. User connects wallet (WalletConnect) — proves ownership of the wallet
  2. User signs a message: "I am revoking all passkeys for my account"
  3. Backend verifies signature against stored wallet_address
  4. Backend revokes all passkeys linked to that wallet
  5. User can register a new passkey on a new device
  6. All existing JWTs from revoked passkeys are invalidated
```

## Migration Path

### Phase 1: Passkey as opt-in (no breaking changes)
- Add passkey registration and login endpoints
- Keep SIWE as the default
- Users can optionally link a passkey and use it for login
- JWT from passkey has same claims as JWT from SIWE

### Phase 2: Passkey as default for returning users
- After successful passkey login, show passkey as primary option
- SIWE remains available for new devices and as fallback
- Analyze metrics: how many users adopt passkey?

### Phase 3: Passkey as sole authentication for daily operations
- SIWE remains only for wallet linking and financial transactions
- New users are guided to register a passkey after wallet connection
- Recovery flow is implemented and tested

## Acceptance Criteria

- [ ] A user can register a passkey after connecting their wallet
- [ ] The wallet signature for linking includes the passkey public key
- [ ] A user can log in with passkey (PIN/fingerprint/pattern) without connecting wallet
- [ ] JWT issued from passkey authentication allows:
  - [ ] Viewing courses and guides
  - [ ] Viewing and editing profile
  - [ ] Submitting crosswords
  - [ ] Claiming UBI (if KYC verified)
  - [ ] Receiving scholarships (if KYC verified)
- [ ] KYC via self.xyz operates independently (returns wallet_address, not user_id)
- [ ] Users without KYC can use the platform but cannot claim scholarships or UBI
- [ ] User-initiated financial operations (donations, premium payments) still require wallet via WalletConnect
- [ ] A recovery flow exists for users who lose their passkey device
- [ ] Multiple passkeys can be registered per user (for multiple devices)
- [ ] Existing SIWE authentication continues to work during migration
- [ ] Passkey authentication works in PWA standalone mode

## Open Questions

1. **Passkey naming**: What user-friendly name should we use? "Passkey", "Biometric Login", "Quick Login"?
2. **Auto-renewal**: Should JWT auto-renew while passkey is present on device, or require manual re-authentication?
3. **Multiple wallets**: Should one passkey be linkable to multiple wallets, or strictly 1:1?
4. **Cross-device sync**: Should we support passkeys synced via iCloud Keychain / Google Password Manager, or require per-device registration?
