# Requirement

**ID**: R-#140
**Title**: Replace SIWE with wallet-native on-chain authentication — single unified flow that works in MiniPay and any other wallet.
**Date**: May 2026 — Redesigned July 2026

---

## Objective

Replace SIWE (`personal_sign`) as the authentication method with an **on-chain nonce verification** that works identically in **every wallet** — MiniPay, OneKey, OKX, MetaMask, Valora — because it only requires a standard transaction (`transfer`), not `personal_sign`.

---

## Motivation

- **SIWE uses `personal_sign`** → MiniPay does NOT support it → learn.tg is completely inaccessible from the Celo reference wallet (14M+ users).
- **MiniPay is the Celo reference wallet.** Opera is expanding country coverage. Sierra Leone may be added soon (already requested repeatedly, precedent: OKX added SL after persistent requests).
- **Passkeys (WebAuthn) require browser APIs** → won't work in MiniPay's embedded WebView consistently.
- **A single unified flow** avoids maintaining two code paths (SIWE for some wallets, something else for MiniPay).
- **On-chain auth works everywhere**: any wallet that can send a transaction can authenticate.

---

## Design Decision: Two Alternatives Presented

### Alternative 1 (Recommended): Single Unified On-Chain Flow

One flow. Every wallet. No exceptions. The user sends a `transfer` transaction with a nonce in the `data` field → backend verifies on-chain → JWT issued.

```
                                     ┌─────────────────────┐
  ┌──────────┐                       │                     │
  │ Any      │  transfer(0 value)    │  Backend monitors   │
  │ Wallet   │  data = nonce         │  Transfer event     │
  │          │ ─────────────────────►│                     │
  │ MiniPay  │                       │  Verifies:          │
  │ OneKey   │                       │  msg.sender == addr │
  │ OKX      │                       │  nonce is valid     │
  │ MetaMask │  ◄────────────────────│                     │
  │ Valora   │  JWT session          │  Issues JWT         │
  └──────────┘                       └─────────────────────┘
```

**How it works:**
1. User connects wallet → frontend detects `window.ethereum` (all wallets expose it)
2. Frontend calls `POST /api/auth/nonce` with `walletAddress` → backend returns a `nonce` (random uint256)
3. Frontend prompts wallet to send: `{ to: backendAddress, value: 0, data: encodeNonce(nonce) }`
   - MiniPay: supports `eth_sendTransaction` with data
   - Every other wallet: supports the same
4. Backend monitors for incoming transactions to `backendAddress` with valid `data`
5. Backend verifies: `msg.sender === walletAddress` AND `nonce` matches AND `nonce` not expired (5 min)
6. Backend issues JWT → session started

**Why this is better:**
- ✅ Works in MiniPay — only needs `eth_sendTransaction`, no `personal_sign`
- ✅ Works in every wallet — OneKey, OKX, MetaMask, Valora, WalletConnect
- ✅ Single code path — no MiniPay special case
- ✅ Gas cost ~$0.0005 on Celo (negligible)
- ✅ User already has CELO for gas (learn.tg UBI gives 0.2 CELO daily)
- ✅ No contract deployment needed — just `transfer` to backend EOA address
- ✅ No WebAuthn dependency — no browser API requirements
- ✅ Works in any WebView, PWA, or embedded browser

**The nonce contract is NOT needed.** The backend EOA receives the transaction. The `data` field carries the nonce. `msg.sender` is verified by the blockchain itself. The backend just watches for incoming transactions with `eth_getTransactionByHash` and valid data.

### Alternative 2: Dual Flow (On-Chain for MiniPay + Passkeys for Others)

Two code paths:
- **Path A (MiniPay):** On-chain nonce via `transfer` with data (same as Alternative 1)
- **Path B (Other wallets):** Passkeys (WebAuthn) — device PIN/fingerprint, no wallet needed after initial linking

This adds complexity: two auth code paths, two DB schemas, two recovery flows. Only consider if passkey UX (no transaction confirmation) proves significantly better than on-chain auth for non-MiniPay users.

## Alternative 1: Detailed Design (Recommended)

### 1. Authentication Flow

```
## First Time (or Session Expired)

  1. User opens learn.tg
  2. Frontend detects wallet is connected (window.ethereum.selectedAddress)
  3. User clicks "Sign In"
  4. Frontend → POST /api/auth/nonce  { walletAddress }
  5. Backend generates nonce = random uint256, stores { walletAddress, nonce, expiresAt }
  6. Backend responds { nonce }
  7. Frontend encodes nonce as hex in data field:
     data = "0x" + nonce.toString(16).padStart(64, '0')
  8. Frontend calls wallet:
     eth_sendTransaction({
       to: NEXT_PUBLIC_ADDRESS,  // backend wallet
       value: "0x0",
       data: data
     })
  9. User confirms in wallet (shows "0 CELO to learn.tg" — clearly a sign-in, not a payment)
  10. Wallet returns txHash
  11. Frontend → POST /api/auth/verify  { walletAddress, txHash, nonce }
  12. Backend:
      a. Fetches transaction by hash (eth_getTransactionByHash)
      b. Verifies tx.to === NEXT_PUBLIC_ADDRESS
      c. Verifies tx.from.toLowerCase() === walletAddress.toLowerCase()
      d. Verifies tx.data contains the correct nonce
      e. Verifies nonce is not expired
      f. Verifies nonce not already used (replay protection)
      g. Marks nonce as used (DELETE or flag consumed)
  13. Backend issues JWT: { sub: user_id, wallet: walletAddress, iat, exp }
  14. Frontend stores JWT → all API calls authenticated

## Session Active (JWT valid)

  - No transaction needed
  - JWT sent in Authorization header
  - touchSession() on each navigation (activity-based expiry, 7 days — per R-#167)

## Session Expired

  - Same as "First Time" flow above
  - User sends a new 0-value transaction with a fresh nonce
  - ~5 seconds, one confirmation click in wallet
```

### 2. Nonce Management

```
Nonce format: random 32-byte uint256 as hex (64 chars)

Storage (in-memory or Redis, NOT database — no persistence needed):
{
  walletAddress: string,
  nonce: string,
  expiresAt: timestamp (5 min from creation)
}

Cleanup: expired nonces removed on access or by periodic sweep (every 10 min)

Replay protection:
  - Each nonce is single-use
  - After successful verification, nonce is deleted
  - txHash is logged in userevent (auth_nonce_used) for audit
```

### 3. Wallet Detection

The same detection logic works for all wallets — MiniPay exposes `window.ethereum` just like MetaMask:

```typescript
// lib/wallet-detect.ts
export function getConnectedAddress(): string | null {
  const eth = (window as any).ethereum
  if (!eth) return null
  return eth.selectedAddress || null
}

export function isMiniPay(): boolean {
  return (window as any).ethereum?.isMiniPay === true
}

export function isOneKey(): boolean {
  return (window as any).ethereum?.isOneKey === true
}
```

The auth flow is identical regardless of wallet type. No branching needed.

### 4. Data Encoding

```typescript
// Frontend: encode nonce into data field
function encodeNonce(nonce: string): string {
  // nonce is a decimal string (e.g., "12345678901234567890")
  const hex = BigInt(nonce).toString(16)
  return '0x' + hex.padStart(64, '0')
}

// Backend: decode nonce from transaction data
function decodeNonce(data: string): string {
  // data is "0x" + 64 hex chars
  const hex = data.slice(2).replace(/^0+/, '') || '0'
  return BigInt('0x' + hex).toString()
}
```

### 5. Backend Verification

```typescript
// app/api/auth/verify/route.ts
export async function POST(request: Request) {
  const { walletAddress, txHash, nonce } = await request.json()

  // 1. Verify nonce exists and is valid
  const stored = nonceStore.get(walletAddress.toLowerCase())
  if (!stored || stored.nonce !== nonce || Date.now() > stored.expiresAt) {
    return Response.json({ error: 'Invalid or expired nonce' }, { status: 401 })
  }

  // 2. Fetch transaction from chain
  const tx = await publicClient.getTransaction({ hash: txHash as `0x${string}` })
  if (!tx) return Response.json({ error: 'Transaction not found' }, { status: 401 })

  // 3. Verify transaction details
  if (tx.to?.toLowerCase() !== NEXT_PUBLIC_ADDRESS.toLowerCase()) {
    return Response.json({ error: 'Invalid destination' }, { status: 401 })
  }
  if (tx.from.toLowerCase() !== walletAddress.toLowerCase()) {
    return Response.json({ error: 'Sender mismatch' }, { status: 401 })
  }

  // 4. Verify nonce in data field
  const decodedNonce = decodeNonce(tx.data || '0x')
  if (decodedNonce !== nonce) {
    return Response.json({ error: 'Nonce mismatch' }, { status: 401 })
  }

  // 5. Consume nonce (replay protection)
  nonceStore.delete(walletAddress.toLowerCase())

  // 6. Look up or create user
  const user = await findOrCreateUser(walletAddress)

  // 7. Issue JWT
  const token = await signJWT({ sub: user.id, wallet: walletAddress })

  // 8. Update billetera_usuario.token (backward compatibility)
  await db.updateTable('billetera_usuario')
    .set({ token, updated_at: new Date() })
    .where('LOWER(billetera)', '=', walletAddress.toLowerCase())
    .execute()

  // 9. Log event
  await recordEvent('auth_nonce_verified', { walletAddress, txHash })

  return Response.json({ token, user })
}
```

## Database Changes

### Modification to `billetera_usuario`

```sql
-- Add auth_method column to track which method was used
ALTER TABLE billetera_usuario ADD COLUMN auth_method TEXT DEFAULT 'siwe';
-- Values: 'siwe' (legacy), 'onchain' (new unified flow)
```

### New Table: `auth_nonce` (optional — for persistent nonce tracking across server restarts)

```sql
-- Only needed if nonce store must survive server restarts
-- Otherwise use in-memory Map (simpler, no migration)
CREATE TABLE authnonce (
    wallet_address VARCHAR(42) PRIMARY KEY,
    nonce TEXT NOT NULL,
    expires_at TIMESTAMP NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**Design preference:** Use in-memory `Map` for nonces. No database table needed. If server restarts, users just request a fresh nonce — the old transactions would fail verification anyway.

## Migration Path

### Phase 1: On-chain auth as opt-in (keep SIWE)
- Add `/api/auth/nonce` and `/api/auth/verify` endpoints
- Frontend detects wallet type — offers on-chain auth button alongside SIWE
- Keep SIWE for backward compatibility
- JWT from on-chain auth has same claims as JWT from SIWE

### Phase 2: On-chain auth as default
- Default to on-chain auth for all wallets
- SIWE available as fallback
- Analyze metrics

### Phase 3: Remove SIWE
- SIWE endpoints deprecated and removed
- `billetera_usuario.token` still used for JWT (no schema change needed)
- `billetera_usuario.auth_method` set to `'onchain'`

---

## Alternative 2: Dual Flow (On-Chain + Passkeys)

If on-chain auth alone proves too slow (~5s block time) compared to passkeys (instant), a dual approach:

| Wallet | Auth Method | UX |
|--------|-------------|-----|
| MiniPay | On-chain nonce | Send 0-value tx, ~5s |
| OneKey / OKX / MetaMask | Passkeys (WebAuthn) | PIN/fingerprint, instant |

**Trade-off:** Two code paths to maintain. Passkeys require WebAuthn API + SPHINCS+ or ECDSA key management (#153 integration). Only pursue if on-chain auth latency is a real problem in practice.

## Security Considerations

| Threat | Mitigation |
|---|---|
| **Replay attack** | Each nonce is single-use; deleted after verification |
| **Nonce guessing** | 256-bit random nonce (practically unguessable) |
| **MITM intercepts txHash** | Attacker can't forge `msg.sender` — blockchain verifies it |
| **Expired nonce reuse** | Nonce expires after 5 minutes |
| **Gas cost abuse** | learn.tg provides 0.2 CELO daily UBI — covers auth transactions |
| **Transaction not confirmed** | Backend waits for at least 1 confirmation before issuing JWT |
| **Front-running** | Nonce is random; even if front-run, attacker can't authenticate as user (tx.from is verified) |

## Acceptance Criteria

- [ ] User can request a nonce via `POST /api/auth/nonce`
- [ ] User can send a 0-value transaction to `NEXT_PUBLIC_ADDRESS` with nonce in data field
- [ ] Backend verifies the on-chain transaction: `to`, `from`, `data` nonce
- [ ] Backend issues JWT after successful verification
- [ ] Works in MiniPay (no `personal_sign` required)
- [ ] Works in OneKey, OKX, MetaMask, Valora (same flow)
- [ ] Works via WalletConnect (same flow — `eth_sendTransaction`)
- [ ] Nonce is single-use (replay protection)
- [ ] Nonce expires after 5 minutes
- [ ] JWT has activity-based expiry (7 days inactivity — per R-#167)
- [ ] Existing SIWE continues to work during migration (Phase 1)
- [ ] Auth events logged in `userevent` (`auth_nonce_requested`, `auth_nonce_verified`, `auth_nonce_expired`)
- [ ] Transaction confirmation shown to user ("Signing in... waiting for confirmation")
- [ ] Works in PWA standalone mode
- [ ] Works in MiniPay embedded WebView
- [ ] Frontend handles pending/confirmed/failed transaction states gracefully
