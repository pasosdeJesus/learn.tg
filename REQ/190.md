# R-#190: Administrative Interface for Verification and Church Management

Enable pdJ verifiers to manage user verification, church data, and pastor bonuses through a secure admin interface accessible only to the verifier wallet.

---

## Objective

Provide a dedicated administrative interface for the verifier (pdJ staff) to:

1. **Manage user verification:** Review profiles, schedule interviews, mark verification fields (`verified_whatsapp`, `verified_telegram`, `verified_email`, `verified_city_id`, `verified_place_of_worship`, `verified_church_relationship`).
2. **Manage church data:** Edit church information, merge duplicate churches, delete churches, assign/transfer pastor role.
3. **Award pastor bonuses:** Grant 44 SLEARN to verified pastors in Sierra Leone and Colombia.
4. **Track verification status:** View pending, scheduled, and completed verifications.
5. **View recent activity:** See newest users and churches (chronological reverse).
6. **Manage personal calendar:** View and block time slots via Radicale integration.

---

## Motivation

Currently, verification is a manual process handled via WhatsApp/Telegram. There is no central interface for the verifier to:

- View pending verification requests
- Mark fields as verified
- Correct user or church data
- Track which pastors have been verified and received bonuses
- Remove duplicate churches
- Manage their calendar of availability

This REQ provides a secure, wallet-based administrative interface to streamline the verification workflow and ensure data integrity.

## Verification Flows

### Flow A: Pastor Registration & Verification

```
1. User registers on learn.tg
   ↓
2. Completes profile: name, country, religion (Christian), city
   ↓
3. Declares church relationship: "pastor"
   ↓
4. Declares church:
   - Church name
   - Pastor name (read-only, auto-filled from user's nombre)
   - Pastor WhatsApp
   - City
   - Denomination
   - Registration number
   - Registration photo
   ↓
5. Requests verification interview (R-#188)
   - Selects available slot → proposed_date_of_interview set
   - Gains access to church dialog (can edit while proposed_date_of_interview is pending)
   ↓
6. If proposed_date_of_interview passes and role is not verified:
   - User loses access to church dialog
   - Must request a new interview
   ↓
7. Verifier reviews profile data and marks verified fields:
   - verified_whatsapp
   - verified_telegram
   - verified_email
   - verified_city_id
   - verified_place_of_worship
   - `verified_church_relationship` VARCHAR(20) — 'pastor' | 'leader' | 'member' | null. Set when verified by verifier.
   ↓
8. If needed, Interview with verifier (optional for pastors already known to verifier):
   - During interview, verifier marks fields as verified
   - conducted_date_of_interview set
   ↓
9. Profile score recalculated: +68 pts (or more, per R-#152 formula)
   - When ALL fields are marked verified, 44 SLEARN awarded as pastor bonus
   - User is encouraged to join Global Disciples course (paid with SLEARN)
   - Full cost in SL, partial elsewhere
   ↓
10. Pastor can now:
    - Administer their church
    - Create/join clusters (R-#189) once they pay for the course
```

### Flow B: Member/Leader Registration & Verification

```
1. User registers on learn.tg
   ↓
2. Completes profile: name, country, religion (Christian), city
   ↓
3. Declares church relationship: "leader" or "member"
   ↓
4. Selects existing church OR declares church:
   - Church name
   - Pastor name of the church
   - Pastor WhatsApp
   - City
   - Denomination
   (No registration fields — only pastors need them)
   ↓
5. Requests verification interview (R-#188)
   - Selects available slot → proposed_date_of_interview set
   ↓
6. Does NOT get access to church dialog
   ↓
7. Verifier reviews profile data and marks verified fields:
   - verified_whatsapp
   - verified_telegram
   - verified_email
   - verified_city_id
   - verified_place_of_worship (if confirmed by pastor)
   - verified_church_relationship = 'leader' or 'member' (if confirmed by pastor)
   ↓
8. If needed, Interview with verifier (optional for people already known to verifier):
   - During interview, verifier marks fields as verified
   - Emphasize: invite pastor within 3 days — we will contact him/her
   - Teach the member to help the pastor register on the platform
   - Encourage pastor to take Web3 & UBI course and claim learn.tg UBI
   - Encourage member to do crossword puzzles in courses with available scholarships
   - Encourage both to join Global Disciples course
   ↓
9. Profile score recalculated: +68 pts (or more, per R-#152 formula)
   ↓
10. 3 days later:
    - Contact the person who provided the data
    - Confirm if the person is the pastor or a member
    - If member, ask about the pastor
    - Introduce the platform
    - Ask if their church is already in a Global Disciples cluster
    - Propose a videoconference after the pastor has met with the member
    - Pastor should have been shown how the platform works
    - Emphasize importance of creating wallet and joining
```

## API Endpoints

### Admin User Management

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/admin/users/recent` | GET | Latest 20 users (chronological reverse) |
| `/api/admin/users` | GET | List users with filters (pending, verified, country, etc.) |
| `/api/admin/user/[id]` | GET | Full user profile with verification status |
| `/api/admin/user/[id]` | PATCH | Update user fields (verified_*, role, interview dates, name, email, whatsapp, telegram, passport_name, passport_nationality, country, city, church_id) |

### Admin Church Management

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/admin/churches/recent` | GET | Latest 20 churches (chronological reverse) |
| `/api/admin/churches` | GET | List churches with filters |
| `/api/admin/church/[id]` | GET | Full church data |
| `/api/admin/church/[id]` | PATCH | Edit church data |
| `/api/admin/church/[id]` | DELETE | Soft-delete church |
| `/api/admin/church/merge` | POST | Merge two churches (duplicate resolution) |

### Admin Pastor Bonus

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/admin/pastor-bonus` | GET | List pastors eligible for 44 SLEARN bonus |
| `/api/admin/pastor-bonus` | POST | Award 44 SLEARN to a specific pastor |

### Admin Calendar (Radicale Integration)

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/admin/calendar/events` | GET | Next 7 days of verifier calendar |
| `/api/admin/calendar/block` | POST | Block a time slot (single or recurring via RRULE) |
| `/api/admin/calendar/unblock` | DELETE | Unblock a time slot |

#### Calendar Configuration

The availability API (`GET /api/verification/availability`) must respect:

1. **Working hours**: Configurable per verifier. Default: Mon-Fri 08:00-16:00 US Eastern (America/New_York), Sat 09:00-12:00. Stored as `working_hours` JSONB on verifier profile or church-level config.
2. **Timezone**: Uses the verifier's church/department timezone from `msip_departamento.timezone` (added in `20260722150000` migration). If not set, defaults to `Africa/Freetown`.
3. **Blocked slots**: Any CalDAV event overlapping a generated slot removes it from availability. This includes:
   - Interview bookings (created by `POST /api/verification/book`)
   - Manual blocks from admin dashboard
   - External events synced to Radicale
4. **Recurring events**: `createEvent()` in `lib/caldav.ts` should support `RRULE` for:
   - Annual: `RRULE:FREQ=YEARLY` (birthdays)
   - Weekly blocks: `RRULE:FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR`
   - Vacation ranges: multi-day events with `DTSTART`/`DTEND`
5. **Timezone display**: Events stored in UTC. Verifier sees them in their configured timezone. Users scheduling interviews see slots in their church/department timezone (from msip_departamento.timezone, implemented in 20260722150000).

#### Timezone Example

| Event | Freetown (GMT+0) | Colombia (GMT-5) |
|---|---|---|
| Interview booked | 3:00 PM | 10:00 AM |
| Verifier sees in calendar | — | 10:00 AM |
| User sees on profile | 3:00 PM | — |

## Component Structure

> **Implementation note:** Components are currently inline in `app/[lang]/admin/page.tsx` as a single `Widget` component. The planned separate components (`AdminCalendar`, `RecentUsers`, `RecentChurches`, `PendingVerifications`) in `components/admin/` are not yet extracted.

```tsx
// app/[lang]/admin/page.tsx
'use client'

import { useAuthAddress } from '@/lib/hooks/useAuthAddress'
import { AdminCalendar } from '@/components/admin/AdminCalendar'
import { RecentUsers } from '@/components/admin/RecentUsers'
import { RecentChurches } from '@/components/admin/RecentChurches'
import { PendingVerifications } from '@/components/admin/PendingVerifications'

export default function AdminDashboard() {
  const { address } = useAuthAddress()
  const verifierWallet = process.env.NEXT_PUBLIC_VERIFIER_WALLET

  if (!address || address.toLowerCase() !== verifierWallet?.toLowerCase()) {
    return <div>Access denied. Verifier wallet required.</div>
  }

  return (
    <div className="container mx-auto py-8 px-4">
      <h1 className="text-2xl font-bold mb-6">🛡️ Verification Dashboard</h1>
      <div className="grid grid-cols-1 lg:grid-cols-2 gap-6">

        </div>
        <div className="space-y-6">
          <RecentUsers limit={10} />
          <RecentChurches limit={10} />
        </div>
      </div>
    </div>
  )
}
```
